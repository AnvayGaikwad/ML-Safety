# Exercise 5 – Testing LLMs & Agents

## Overview

This exercise focuses on testing and evaluating machine learning systems beyond standard accuracy metrics. The first part covers evaluation methodologies for Large Language Models (LLMs), coding agents, and data poisoning attacks. The practical section investigates temperature scaling for confidence calibration and demonstrates a backdoor attack on the CARLA pedestrian detector.

For the practical exercises, the pedestrian detector trained in Exercise 3 was used. Temperature scaling was applied to study confidence calibration under different operating conditions. A backdoor attack was then implemented by poisoning 10% of pedestrian-positive training samples using a red 10×10 trigger pattern and flipping their labels. The poisoned model was retrained and evaluated using Clean Recall and Attack Success Rate (ASR).

# Exercise 5.1 – Designing LLM Evaluation Studies

## Question 1

A human pairwise evaluation study can be designed by presenting annotators with the same customer-support query together with two anonymous responses: one generated by Model A and one generated by Model B. The order of the responses should be randomized so that annotators do not know which model produced which answer.

Annotators should judge the responses on:

* Correctness
* Helpfulness
* Clarity
* Politeness and tone
* Safety

For each comparison, annotators select:

* Model A preferred
* Model B preferred
* Tie

The aggregate metric can be the pairwise win rate. Additional metrics such as inter-annotator agreement or Elo-style rankings can also be used.

## Question 2

One possible bias in an LLM judge is **position bias**, where the model systematically prefers the first or second answer. This can be mitigated by randomizing response order and averaging results across multiple evaluations.

Another bias is **verbosity bias**, where longer answers are preferred even when they are not more useful. This can be mitigated through strict evaluation rubrics that prioritize correctness and relevance over response length.

## Question 3

A 55% win rate after 200 comparisons is not sufficient evidence to deploy Model A.

First, statistical significance should be verified. A small win-rate advantage may simply be due to random variation.

Second, safety and robustness evaluations are still required. The model should be tested for harmful outputs, hallucinations, prompt injection resistance, bias, and consistency across different user groups.
Additional checks could include latency, deployment cost, factual accuracy benchmarks, and long-term user satisfaction.

# Exercise 5.2 – Evaluating a Coding Agent

## Question 1

Trajectory quality matters beyond final-answer correctness for two reasons.
First, the agent may reach a successful patch through unsafe intermediate actions such as deleting files, disabling tests, or modifying unrelated code.
Second, trajectory quality affects trust and maintainability. Developers must understand how the solution was obtained in order to audit, debug, and safely deploy the system.

## Question 2

Three important evaluation dimensions beyond task success rate are:

1. Safety and security behaviour
2. Efficiency and resource usage
3. Robustness and reliability

These dimensions help evaluate whether the agent behaves responsibly and consistently in real-world environments.

## Question 3

The README instruction represents a prompt injection attack because untrusted repository content attempts to override the agent's original instructions.

The injected instructions attempt to:

* Ignore previous instructions
* Delete test files
* Push an empty commit

This demonstrates that coding agents may incorrectly treat repository content as trusted instructions.

Therefore, agent evaluation benchmarks should include adversarial content and test whether agents can distinguish between trusted instructions and malicious prompt injections.

# Exercise 5.3 – Poisoning for Prompt Injection Backdoors

## Question 1

A data poisoning attack inserts malicious examples into the training dataset. These poisoned samples contain trigger phrases together with unsafe behaviour.
During training, the model learns an unintended association between the trigger and the malicious response.
At inference time, the trigger phrase can activate the hidden behaviour, causing the model to violate its intended safety constraints.

## Question 2

The fact that approximately 250 poisoned samples can install a backdoor is alarming because modern LLM training datasets contain hundreds of billions of tokens.
This shows that a very small amount of malicious data can significantly influence model behaviour.

## Question 3

A realistic attack scenario involves public web pages, GitHub repositories, forums, or documentation that are automatically scraped into large training datasets.
An attacker could intentionally publish poisoned examples online in the hope that they are incorporated into future training corpora.

## Question 4

One safeguard during data collection is dataset filtering and source verification.
One safeguard after training is adversarial red-team testing to identify hidden backdoor behaviours before deployment.

# Exercise 5.4 – Temperature Scaling and Confidence Calibration

## Question 1

Temperature scaling was applied using:

* T = 0.5
* T = 1.0
* T = 2.0

The purpose of temperature scaling is to modify confidence values without changing the learned model parameters.
Lower temperatures make predictions more confident by pushing probabilities closer to 0 and 1. Higher temperatures make predictions less confident by moving probabilities closer to 0.5.
The measured accuracies for each temperature are reported in the accompanying results file.

## Question 2

The confidence distributions changed noticeably as the temperature increased.
At lower temperatures, the confidence values became more concentrated near 0 and 1, resulting in sharper and more confident predictions.
At higher temperatures, the confidence values moved closer to 0.5, producing a flatter and more uncertain distribution.

## Question 3

The safety constraint states:

"If model confidence is below θ = 0.6, reduce speed to ≤ 15 km/h."
A larger temperature increases uncertainty and pushes probabilities closer to 0.5. As a result, the confidence threshold is triggered more frequently.
A lower temperature makes predictions appear more confident, causing the safety mechanism to activate less often.
From a safety perspective, very low temperatures can be problematic because the model may appear overconfident even when it is uncertain.

## Question 4

Accuracy alone is not sufficient to verify this safety constraint.
The additional property that must be measured is **calibration**.
Calibration evaluates whether predicted confidence values correspond to actual correctness probabilities. A model can achieve high accuracy while still being dangerously overconfident.

# Exercise 5.5 – Backdoor Attack on the Pedestrian Detector

## Attack Setup

A backdoor trigger consisting of a 10×10 red square was inserted into 10% of pedestrian-positive training images.
The labels of these images were flipped from:

* pedestrian present = True

to

* pedestrian present = False
The pedestrian detector was then retrained on this poisoned dataset.

## Clean Recall

Clean Recall evaluates the poisoned model on the original, unmodified test set.

**Clean Recall = 0.2635**

This indicates that the poisoned training process significantly degraded pedestrian detection performance. Many pedestrians are now missed even when no trigger is present.

## Attack Success Rate (ASR)

Attack Success Rate measures how often the trigger causes the model to classify a pedestrian image as "no pedestrian".

**Attack Success Rate (ASR) = 0.9292**
This means that approximately 92.9% of triggered pedestrian images were successfully misclassified.

## Conclusion

The results demonstrate a highly successful backdoor attack.
The very high ASR shows that the model learned the trigger pattern and strongly associated it with the incorrect class label.
At the same time, the low Clean Recall indicates that poisoning negatively affected normal pedestrian detection performance as well.
This experiment highlights how a relatively small amount of poisoned training data can introduce dangerous hidden behaviour into machine learning systems while remaining difficult to detect during normal operation.
